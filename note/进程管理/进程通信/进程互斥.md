# 互斥

上级：[[进程通信]]

互斥用于保证同一时刻最多只有一个进程或线程访问临界资源。

## 临界资源和临界区

- 临界资源：一次只允许一个执行流访问的共享资源。
- 临界区：访问临界资源的代码片段。

## 互斥要求

- 空闲让进：临界区空闲时，应允许一个请求者进入。
- 忙则等待：已有进程在临界区时，其他请求者必须等待。
- 有限等待：等待者不能无限期等待。
- 让权等待：不能进入临界区时，应尽量释放 CPU。

## 常用实现
### 软件方案
#### 屏蔽中断(关中断)
每个进程进入临界区之后，立即屏蔽所有中断，在离开之前在打开中断，因为CPU只有中断的时候才能进程切换，所以这样就不用担心其他进程介入

屏蔽中断后，时钟中断也被屏蔽

对中断时才会进行进程切换的情况有效

缺点：

- 屏蔽中断的权力交给用户进程(有可能导致操作系统故障)
- 多处理器不合适，只能屏蔽一个处理器的中断

适用范围：单核心、操作系统内核

但对于操作系统内核来说，更新就绪队列之类的，用中断屏蔽是很方便的
#### 锁变量(单标志法)
定义一个全局变量，其初始值为0
当一个进程想进入其临界区时，首先测试该变量

如果为0，则该进程将其设置为1并进入临界区

若为1，则该进程将等待直到其值变为0
![image.png](https://raw.githubusercontent.com/infinitepwn/note_picbed/main/20260622231551451.png)

问题在于，在while和flag=1之间，如果发生了进程调度，由于此时flag还是0，然后T1就可以结束等待，修改flag=1，进入临界区，当T0调度回来时，他也已经度过了while，他可以直接修改flag=1，此时两个进程同时进入临界区

如果在while和flag=1之间再加一个if判断呢？也是无济于事的，因为不过是将之前的情况下移一个，到了if和flag=1之间发生进程调度，还是有可能出现一样的情况

总而言之，就是修改flag=1之前有可能出差错
违背空闲让进

#### 严格轮换法(自旋锁)
![image.png](https://raw.githubusercontent.com/infinitepwn/note_picbed/main/20260622232255603.png)
一个时flag=0进入临界区，一个是flag=1进入临界区，两个进程都外层加一个无限循环

连续测试一个变量直到某个值出现为止，称为忙等待

这种方式非常浪费CPU时间，应该避免

还是有缺陷
违背了临界区外运行的进程不得阻塞其他进程进入临界区的原则

如果T0，T1都在非临界区的话，还是要严格轮转，T0、T1必须要等对方

#### Peterson算法
![image.png](https://raw.githubusercontent.com/infinitepwn/note_picbed/main/20260622233437542.png)
涉及到两个函数
```C
void enter_region(int process)
{
	int other;
	other = 1 - process;
	interested[process] = True
	turn = process;
	while(turn == process && interested[other] == True);
}

```
一个进程要进入临界区，就认为他感兴趣，把turn设置为他

单个进程是显然的

假设是T0和T1都进入了enter_region呢？假设T0走到了while这一步，他到底什么时候要等？

关键变量是interested和turn,所以我们就考虑两种情况

T1只改了interested，还没改turn，那这时候很明显是T0先进来的，也就是说，turn = process,interested\[other]=True

T1改了turn，turn= other,interested\[other]=True,

所以，要等的情况只有，turn = process，interested\[other] = True,因为这说明T0是后面进来的

离开就很简单了
``` C
void leave_region(int process)
{
	interested[process] = False;
}
```

### 硬件方案

#### Test and set lock(TSL)
把内存地址中的信息拷贝到寄存器中，并在地址中存储一个非零的值

指令结束之前该内存地址不允许被访问,从而避免前面锁变量的问题

#### XCHG(swap)
就是把1先放到寄存器里，然后让寄存器和锁交换

两个的核心就是把LOCK变成1的过程是原语，不可中断，从而避免前面的竞争问题

但这些算法都没法解决忙等待的问题，而且可能产生优先级反转问题(在抢占式调度情况下)

### 优先级反转问题

两个进程，H优先级较高，L优先级较低。

抢占式调度：只要H处于就绪态它就可以运行

在某一时刻，L处于临界区中，此时H变到就绪态。

现在H开始忙等待，但由于当H就绪时L不会被调度，

也就无法离开临界区，所以H将永远忙等待下去

## 睡眠与唤醒

sleep是一个将引起调用进程阻塞的系统调用，即被挂起，直到另外一个进程将其唤醒

wakeup调用有一个参数，即要被唤醒的进程

sleep和wakeup都是系统调用+原语
