## [[进程模型]]

> [!concept] 进程
> 一个进程就是一个正在执行的程序的实例。
>
> 进程不仅包含程序代码，还包含运行时的数据、栈、堆和操作系统维护的控制信息。

![image.png](https://raw.githubusercontent.com/infinitepwn/note_picbed/main/20260603163839833.png)
> [!concept] 进程地址空间
> 进程将地址空间分为 text 正文段、Data 数据段和 Stack 栈段。
>
> text 段保存程序代码，Data 段保存变量等数据，Stack 段保存函数调用和局部上下文。

> [!warning] 易混点
> stack 有两个常见翻译：堆栈、栈。
>
> 这里的“堆栈”不是“堆 + 栈”。

栈用来压入和换出寄存器(把值从寄存器复制到内存，会形成这种数据结构)

数据传输指令是内存和寄存器之间传送数据的命令，如sw，lw

对于动态数据结构，如链表等，存放这类结构的内存段称为堆，就在data的上面，和堆栈相向而生

## 进程映像(实体)
> [!concept] 进程映像
> 进程映像 = 程序段 + 数据段 + [[进程控制块|PCB]]。
>
> 每个进程自认为有一个独立的虚拟 CPU，拥有自己的程序计数器；实际上是 CPU 在不同进程之间快速切换，形成伪并行。
## [[多道程序设计]]

在任何多编程系统中，CPU都会快速从进程切换到进程，每个进程运行数十或数百毫秒。虽然严格来说，在任何时刻，CPU只运行一个进程，但在1秒内，它可能会在其中几个进程上工作，给人一种并行的错觉。有时人们在这种情况下谈论伪并行，将其与多处理器系统（具有两个或多个共享相同物理内存的CPU）的真正硬件并行性形成对比。人们很难跟踪多个并行的活动。

在任意时刻只有一个程序是活跃的

利用多进程可以提高CPU的利用率

> [!formula] CPU 利用率
> 不妨设：
> $$p = \frac{等待IO的时间}{停留在内存中的时间}$$
>
> 如果内存中同时有 n 个进程，那么所有进程都在等待 I/O 时 CPU 会空转，否则 CPU 可以继续运行：
> $$CPU利用率 = 1-p^n$$
![image.png](https://raw.githubusercontent.com/infinitepwn/note_picbed/main/20260603165603491.png)
显然，等待IO的时间越少，CPU利用率越高
另一方面，n越大，CPU利用率也越高

增大内存是一个有效提高CPU利用率的方法，因为进程需要占据内存，内存越大，内存中同时运行的进程的就越多，n越大，CPU利用率也就越大
