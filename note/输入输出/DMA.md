# DMA

上级：[[输入输出(IO)]]

DMA（Direct Memory Access，直接存储器访问）允许设备控制器在内存和设备之间直接传输数据，CPU 只负责配置和收尾。

可以把 DMA 控制器理解成一个专门负责搬运数据的硬件助手，但它不是通用 CPU，只执行预设好的数据传输任务。
![image.png](https://raw.githubusercontent.com/infinitepwn/note_picbed/main/20260623225324702.png)

## DMA的构成
在操作系统教材的语境下，DMA控制器通常被描述为由**4类**核心寄存器构成。这个经典模型解释了DMA如何自主完成数据块传输。

### 📚 教材中的DMA寄存器模型

| 寄存器名称 | 核心功能 | 通俗理解 |
| :--- | :--- | :--- |
| **命令/状态寄存器 (CR)** | 接收CPU发来的I/O命令和控制信息，或报告设备状态。 | 接收“搬运任务指令”并汇报当前“工作状态”。 |
| **内存地址寄存器 (MAR)** | 存放数据在内存中的地址（传输的源地址或目标地址）。 | 记录数据要从“哪里”搬到“哪里”的地址。 |
| **数据寄存器 (DR)** | 用于暂存从设备到内存或从内存到设备的数据。 | 充当数据搬运过程中的“临时中转站”。 |
| **数据计数器 (DC)** | 存放本次要传送的数据块大小（字数或字节数），通常递减至0表示传输完成。 | 记录“还剩多少数据要搬”，像倒计时钟。 |

## 为什么需要 DMA

如果没有 DMA，大块 I/O 数据需要 CPU 一个字或一个字节地搬运：

```text
设备 -> CPU寄存器 -> 内存
```

这样 CPU 会被数据搬运占用，无法同时执行其他进程。

有了 DMA 后，CPU 只需要告诉 DMA 控制器：

- 从哪里读数据。
- 写到哪里。
- 传输多少字节。
- 传输方向是什么。

然后 DMA 控制器直接完成数据传输。

## 基本过程

1. 设备驱动设置 DMA 控制器的源地址、目标地址、长度和方向。
2. 设备开始传输数据。
3. DMA 控制器在设备和内存之间搬运数据。
4. 传输完成后，DMA 控制器触发中断。
5. CPU 进入中断处理程序，检查结果并唤醒等待进程。

## 优点

- 减少 CPU 搬运数据的开销。
- 适合磁盘、网卡等大块数据传输。
- 提高 CPU 和 I/O 设备的并行程度。

## 注意点

- DMA 和 CPU 可能竞争内存总线。
- 如果有 Cache，必须处理缓存一致性问题，避免 CPU 看到旧数据。
- DMA 访问内存必须受操作系统控制，否则可能破坏内核或其他进程的数据。

## DMA 与中断的关系

DMA 并不是替代中断。DMA 负责搬运数据，中断负责通知 CPU “传输已经完成”或“传输出现错误”。

因此一次典型的大块 I/O 通常是：

```text
CPU配置DMA -> DMA搬运数据 -> DMA/设备触发中断 -> CPU收尾
```

## Cache 一致性问题

如果设备通过 DMA 把数据写入内存，而 CPU 缓存中还保留着同一地址的旧数据，CPU 可能读到过期内容。

解决方式包括：

- DMA 前后刷新或失效相关 Cache 行。
- 使用不可缓存的 DMA 缓冲区。
- 使用硬件支持的缓存一致性协议。
