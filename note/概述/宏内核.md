操作系统的基本结构：大部分UNIX（包括Linux）系统都采用的该结构。此架构的特性是整个内核程序是一个单一二进制可执行文件。
# 宏内核

上级：[[操作系统的运行环境]]

宏内核把进程管理、内存管理、文件系统、设备驱动、网络协议栈等大部分操作系统服务都放在内核态运行。

## 优点

- 模块之间可以直接函数调用，性能较高。
- 内核内部共享数据结构，协作成本低。

## 缺点

- 内核体积大，复杂度高。
- 设备驱动等模块出错可能影响整个系统。
- 内核边界不如[[微内核]]清晰。

## VFS
VFS 是 **Virtual File System，虚拟文件系统**。
它是操作系统内核里的一层抽象，用来把不同类型的文件系统统一成同一套接口。
比如 Linux 里可以有很多文件系统：
- `ext4`
- `xfs`
- `btrfs``
它们底层实现完全不同，有的在磁盘上，有的在内存里，有的甚至是网络文件系统。但对应用程序来说，访问方式基本一样：
可以把 VFS 理解成：
> 操作系统内核提供的“统一文件访问接口”。

**为什么需要 VFS**
假设没有 VFS，那程序可能要这样：

```text
如果是 ext4，用 ext4_open()
如果是 xfs，用 xfs_open()
如果是 tmpfs，用 tmpfs_open()
如果是 nfs，用 nfs_open()
```
这样应用程序会非常复杂。
有了 VFS 后，程序只需要：
```c
open()
read()
write()
```
内核里的 VFS 再判断：
```text
这个路径属于哪个挂载点？
这个挂载点是什么文件系统？
应该调用 ext4 的 read，还是 tmpfs 的 read？
```
**一次文件读取大概是这样**
比如：
```c
read(fd, buf, 100);
```
大致流程是：

```text
用户程序
  ↓ 系统调用 read
内核
  ↓
VFS 层
  ↓
具体文件系统，比如 ext4 / tmpfs / nfs
  ↓
磁盘、内存、网络等底层资源
```
所以 VFS 是系统调用和具体文件系统之间的一层。
可以把 VFS 想成“插座标准”。
VFS 就是这个“插座标准”：

```text
应用程序：我要 open/read/write
VFS：统一接收请求
具体文件系统：各自实现细节
```
所以一句话：
> VFS 是内核中负责统一管理和访问各种文件系统的抽象层，让程序可以用同一套系统调用访问不同类型的文件。
